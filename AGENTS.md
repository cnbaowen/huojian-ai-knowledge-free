# 火建AI知识管理免费版仓库安全规则

## 1. 适用范围与最高原则

- 本文件适用于本仓库根目录及其全部子目录。
- 本项目是独立的免费版项目，不是商业版的分支、补丁集、部署壳或运行时附属组件。
- 所有分析、生成、修改、测试产物和补丁都必须限制在当前 Git 根目录 `E:\huojian-ai-knowledge-free` 内。
- 除非用户明确修改本规则，否则任何便利性、历史实现或临时调试需求都不能突破此边界。

## 2. 写入前的强制边界检查

在修改文件、安装依赖、生成代码、构建镜像或运行会产生文件的命令前，必须：

1. 执行 `git rev-parse --show-toplevel`，确认结果为 `E:/huojian-ai-knowledge-free`。
2. 执行 `git status --short --branch`，记录并保护用户已有改动。
3. 将所有目标路径解析为绝对路径，确认仍位于本仓库根目录内；不得通过 `..`、符号链接、目录联接、挂载或环境变量越界。
4. 对可能写入缓存、日志、临时文件、数据库、构建产物或容器挂载的命令，先确认输出目录也在本仓库内，或位于本项目专属的 Docker 卷中。

如果无法确认路径归属，立即停止，不得试探性写入。

## 3. 商业版隔离红线

免费版补丁不得以任何方式修改、读取后复制、导入、链接或运行依赖商业版仓库及其产物，包括但不限于：

- `E:\enterprise-ai-v2.3.2-trial-ready` 或其他商业版检出目录；
- 商业版源码、Git 历史、分支、标签、补丁、构建目录或发布包；
- 商业版数据库、备份、日志、客户资料、模型配置、密钥、许可证和品牌私有资产；
- 商业版容器、镜像、网络、卷、服务地址、Webhook 或运行中的 API；
- 仓库外构建器、客户交付脚本以及指向商业版的本地路径依赖。

禁止新增 Git submodule、Composer `path` repository、NPM `file:`/workspace 依赖、符号链接、目录联接或脚本复制步骤来间接连接商业版。需要的能力必须在本仓库内独立实现，或使用许可兼容、版本锁定、可公开获取的第三方依赖。

仓库中既有的“商业版同款”文字、样式或历史来源说明不构成继续复制商业版内容的授权。

## 4. 免费版产品边界

允许维护的核心范围是独立知识管理闭环：

- 登录与免费版访问控制；
- 知识分类、文档上传、解析、切片和检索；
- 内部知识问答、引用溯源、会话上下文与反馈；
- 知识质量检查、RAG 评测、模型配置；
- 免费版系统状态与企业微信机器人配置；
- 免费版独立安装、升级、测试、文档和发布流程。

不得引入或恢复商业版专属模块、路由、数据表、菜单、服务或样式资产，包括客户服务、微信客服、CRM、销售企微、内容生产/发布、情报、智能体、OpenClaw、公共网关、许可证管理和系统升级模块。此类能力必须在源码和路由层保持物理缺失，而非仅隐藏菜单或返回 403。

## 5. 运行时和数据隔离

- Compose 项目名保持 `huojian-ai-knowledge-free`。
- 默认端口保持前端 `18080`、后端 `18000`；如需调整，也不得占用或复用商业版端口。
- 网络、卷、数据库名和文档存储必须使用免费版专属名称，禁止声明或连接外部商业版网络和卷。
- 禁止复用、重启、停止、迁移或清理商业版容器及数据。
- 测试只能使用本仓库自带的虚构演示数据或为本任务新建的无敏感数据；不得导入真实客户数据。
- `.env`、密钥、令牌、数据库文件、日志、备份、上传文档和本地产物不得提交 Git。

## 6. 修改与 Git 安全

- 只修改完成当前免费版任务所必需的文件，保留并避开无关的用户改动。
- 禁止使用 `git reset --hard`、`git clean -fd`、`git checkout -- .` 等破坏性命令。
- 未经用户明确要求，不得提交、打标签、推送、创建远程仓库或改写历史。
- 不得把商业版仓库配置为本项目远程，也不得让商业版工作区执行本项目发布命令。
- 不得提交 `vendor/`、`node_modules/`、`dist/`、`output/`、运行日志或临时浏览器数据。

## 7. 每个补丁的最低验证

根据改动范围执行足够的验证，并至少完成：

1. 再次确认 `git rev-parse --show-toplevel` 和 `git status --short`。
2. 检查新增/修改内容不存在仓库外绝对路径、本地路径依赖、商业版服务地址或商业模块路由。
3. 运行与改动相关的后端、前端或容器测试。
4. 发布相关改动运行现有边界/发布检查；Linux 环境执行 `bash scripts/release-check.sh`，Windows 环境应执行等价检查，不得因为缺少 Bash 跳过边界验证。
5. 确认没有新增密钥、客户数据、数据库、备份、日志或构建产物进入 Git。

若发现现有代码违反边界，应先报告具体文件和风险；除非任务明确包含清理，不得借机大范围重写。任何验证失败都必须如实说明，不得把补丁描述为已通过或可发布。

## 8. 冲突处理

当任务要求与本文件冲突时，默认选择更严格的隔离策略并停止越界操作。只有用户明确要求修改免费版边界规则后，才能先更新本文件，再进行相应实现；“参考商业版”“保持同款”或“临时复用”不视为解除隔离的授权。
